## Context

当前 `TxtParser` 仓库还没有形成可开发的 Node.js/TypeScript 包工作区，根目录仅包含许可证与 OpenSpec 文档，尚不具备依赖安装、类型检查、构建与发布的基础设施。与之对应，`../HtmlParser` 已经具备一套可运行的包工程基线，包括包元数据、TypeScript 配置、Rolldown 构建、Jest 测试、ESLint/Prettier 约定、源码入口与发布相关文件。

本次变更的目标不是实现 TXT 解析能力，而是为 `@hamster-note/txt-parser` 建立一个与 `HtmlParser` 对齐的独立工程基线，使后续解析逻辑可以在统一约定下继续演进。约束包括：

- 保持 `TxtParser` 作为独立包存在，不能通过软链接或共享配置文件路径依赖 `HtmlParser`。
- 工程约定尽量与 `HtmlParser` 保持一致，以降低后续多解析器包之间的维护成本。
- 所有对外身份信息都必须切换为 TxtParser 语义，不能残留 HTML 解析命名或行为暗示。
- 当前阶段只处理工作区初始化，不引入任何 HTML/TXT 解析实现细节。

## Goals / Non-Goals

**Goals:**

- 复制并落地 `HtmlParser` 中与包工程初始化直接相关的基础文件、目录结构和脚本约定。
- 将包身份统一切换为 `@hamster-note/txt-parser`，同步更新包名、入口、构建产物声明、文档与发布元数据。
- 为当前仓库建立最小可用的源码入口和构建链路，使其具备 `install`、`build`、`typecheck`、`test`、`publish` 前置准备能力。
- 让未来新增 TXT 解析实现时，可以在不重构工程基线的前提下直接扩展 `src`、测试与发布流程。

**Non-Goals:**

- 不实现任何 TXT 解析、文本切分、模型转换或运行时业务逻辑。
- 不修改 `HtmlParser` 包本身，也不尝试抽象跨仓库共享配置。
- 不在本次变更中重新设计工具链，只复用已验证的现有工程方案。
- 不承诺一次性补齐所有 CI/CD、GitHub 工作流或发布自动化能力，超出本地开发与发布准备范围的内容后续再补。

## Decisions

### 1. 以 `HtmlParser` 作为工程基线来源，采用“选择性复制 + 身份重写”策略

`TxtParser` 将以 `../HtmlParser` 已存在的工程结构作为初始化蓝本，优先复制与包本地开发、构建、测试、发布相关的文件，例如 `package.json`、TypeScript 配置、构建配置、测试配置、格式化/静态检查配置、`.gitignore`、`.npmignore`、`README` 与 `src` 入口骨架。复制后对所有项目身份字段进行统一重写。

这样做的原因是 `HtmlParser` 已经验证过工具链可行性，直接对齐可以减少人工搭建遗漏，并保持 HamsterNote 解析器家族的一致性。

备选方案：从零手工搭建新的轻量工作区。该方案初期文件更少，但会造成脚本、依赖和目录结构与现有解析器包分叉，后续维护成本更高，因此不采用。

### 2. 保留工具链一致性，不在初始化阶段替换构建或测试方案

当前设计默认沿用 `HtmlParser` 中已经使用的 TypeScript + Rolldown + Jest + ESLint + Prettier 组合，并保持相近的脚本命名与输出约定。`TxtParser` 在初始化时不引入新的构建器、测试框架或包管理方案。

这样做可以保证：

- 团队对解析器包的开发体验保持一致；
- 后续迁移或问题排查可以直接参照 `HtmlParser`；
- 当前变更聚焦于工作区初始化，而不是工具链重构。

备选方案：切换到更轻量的测试或构建方案。该方案需要额外验证兼容性，并会扩大本次变更范围，因此暂不采用。

### 3. 提供“可构建但不承诺解析能力”的最小源码入口

初始化后的 `TxtParser` 应包含最小可构建的 `src/index.ts` 入口，以及为未来 TXT 解析实现预留的导出边界，但不会伪造任何未实现的解析能力。源码入口的职责是确保包可以被构建、类型检查和发布，而不是提前定义复杂领域模型。

这样做可以在不引入虚假能力承诺的前提下，确保整个工作区立即可验证。

备选方案：直接复制 `HtmlParser` 的源码实现再逐步删减。该方案存在误带 HTML 语义、测试与依赖耦合的风险，因此不采用。

### 4. 对身份替换采用“集中清单校验”而非逐文件随意修改

实施时应先建立一份身份替换清单，覆盖至少以下位置：

- `package.json` 中的 `name`、`description`、入口与发布字段；
- `README.md` 中的包名、使用说明与示例；
- 构建配置、测试配置与源码入口中的输出名称或路径约定；
- 发布辅助文件、忽略规则与锁文件上下文中的项目名痕迹。

集中校验的目的是避免残留 `HtmlParser`、`html-parser`、HTML 语义文案或错误的导出路径，降低初始化后“能编译但身份错误”的风险。

备选方案：复制后凭肉眼逐步修补。该方案遗漏风险高，不适合作为标准流程，因此不采用。

### 5. 保持本地独立文件，不通过共享配置或链接复用参考工程

虽然 `TxtParser` 与 `HtmlParser` 需要对齐，但最终文件应保存在 `TxtParser` 仓库本地，不通过 `yarn link`、相对路径 extends、软链接等方式直接依赖参考仓库的配置文件。

这样做可以保证 `TxtParser` 单仓库可安装、可构建、可发布，也符合其“独立解析器包”的定位。

备选方案：通过链接复用 `HtmlParser` 配置。该方案会使仓库边界模糊，且在 CI、发布与团队协作中引入隐藏依赖，因此不采用。

## Risks / Trade-offs

- `[复制范围过大]` → 可能引入与 TXT 场景无关的测试、示例或 HTML 特定依赖；通过“仅复制工程基线文件 + 审核领域源码”控制范围。
- `[身份替换遗漏]` → 可能导致发布出的包仍带有 HTML 命名；通过集中清单校验和构建后人工复查产物元数据缓解。
- `[工具链完整复制带来额外依赖]` → 初始化阶段 devDependencies 可能偏多；先接受短期冗余，待 TXT 解析能力稳定后再做收敛。
- `[最小入口设计过于保守]` → 可能让后续实现仍需调整导出边界；通过只固化包入口与构建契约，避免过早锁定内部结构。
- `[参考工程后续继续演进]` → `TxtParser` 的基线可能很快与 `HtmlParser` 再次产生偏差；通过在文档中明确“本次初始化参考当前工程状态”，后续按需同步。

## Migration Plan

1. 盘点 `HtmlParser` 中属于“工程基线”的文件和目录，排除 HTML 领域实现与不必要的历史产物。
2. 将选定的工程文件复制到 `TxtParser`，建立本地源码入口、配置文件与发布元数据。
3. 按统一清单完成项目身份替换，确保包名、文档、脚本和输出约定全部切换到 TxtParser。
4. 执行依赖安装、类型检查、测试与构建，确认初始化后的工作区具备后续开发基础。
5. 如验证失败或发现复制策略不合适，回滚本次新增的工程文件与元数据修改，恢复到仅包含规范文档的初始状态，再重新收敛复制范围。

- 初始化阶段是否需要同时引入占位测试文件，以确保测试链路可运行，还是仅保留测试配置即可？
- `TxtParser` 的初始版本号应完全沿用参考包当前版本节奏，还是以独立初始版本重新开始？
- 若 `HtmlParser` 当前依赖中存在明显与 HTML 强绑定的包，是否应在初始化阶段先保留占位依赖，还是直接裁剪？
